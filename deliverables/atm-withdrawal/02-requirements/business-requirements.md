## 1. 專案概述

### 1.1 專案背景

富邦銀行為提供持卡人 24 小時的自助金融服務，需建置 ATM 提款功能。本文件定義提款服務之業務需求、流程規則與驗收標準，作為開發、測試與驗收的共同依據。

### 1.2 專案目的

- 使持卡人能於 ATM 安全地插卡、驗證密碼並提取現金。
- 確保交易過程中帳戶餘額之正確性與資料一致性。
- 透過 BDD 情境確保需求可被自動化驗證。

### 1.3 專案範圍

| 範圍類別 | 說明 |
|----------|------|
| 包含範圍（In Scope） | 插卡、密碼驗證、提款、餘額扣款、退卡 |
| 排除範圍（Out of Scope） | 存款、轉帳、餘額查詢、跨行交易、鈔票面額組合 |

## 4. 業務流程說明

### 4.1 主要流程（Happy Path）

1. 持卡人於富邦銀行 ATM **插入提款卡**。
2. 系統要求輸入密碼，持卡人輸入密碼。
3. 密碼**驗證通過**，畫面解鎖可供選擇服務。
4. 持卡人**選擇提款 1000 元**。
5. 系統確認**帳戶餘額充足**，出鈔機提供 1000 元現金。
6. 系統於帳戶**扣除 1000 元**。
7. 持卡人**選擇結束服務**，ATM **退出提款卡**。

### 4.2 流程圖（文字版）

```
[插入提款卡]
      │
      ▼
[輸入密碼] ──密碼錯誤──> [顯示「密碼錯誤」] ──> [無法選擇服務]
      │ 密碼正確
      ▼
[選擇服務／提款金額]
      │
      ▼
[檢查餘額] ──餘額不足──> [顯示「餘額不足」] ──> [不出鈔／餘額不變]
      │ 餘額充足
      ▼
[出鈔] ──> [扣款] ──> [結束服務] ──> [退出提款卡]
```

## 5. 業務需求明細

### 5.1 功能性需求（Functional Requirements）

| 需求編號 | 需求名稱 | 需求描述 | 優先級 |
|----------|----------|----------|--------|
| FR-01 | 插卡 | 系統須能接受持卡人插入提款卡並識別對應帳戶 | 高 |
| FR-02 | 密碼驗證 | 系統須驗證輸入密碼，正確方可進入服務選單 | 高 |
| FR-03 | 服務解鎖控制 | 密碼未通過前，系統不得允許選擇任何服務 | 高 |
| FR-04 | 提款 | 系統須在餘額充足時提供指定金額現金 | 高 |
| FR-05 | 扣款 | 提款成功後須即時於帳戶扣除相同金額 | 高 |
| FR-06 | 餘額不足處理 | 餘額不足時須阻擋交易、不出鈔且餘額不變 | 高 |
| FR-07 | 結束服務與退卡 | 持卡人結束服務後須退出提款卡並清除交易狀態 | 高 |

### 5.2 非功能性需求（Non-Functional Requirements）

| 需求編號 | 類別 | 需求描述 |
|----------|------|----------|
| NFR-01 | 安全性 | 密碼須為 4 至 6 位數字，錯誤時不得洩漏餘額資訊 |
| NFR-02 | 資料一致性 | 出鈔與扣款須為一致操作，不得只出鈔未扣款 |
| NFR-03 | 可測試性 | 所有業務規則須可由自動化 BDD 情境驗證 |
| NFR-04 | 可維護性 | 系統須遵循 DDD 分層與 SOLID 原則 |
| NFR-05 | 效能 | 單筆交易回應時間須 ≤ 30 秒 |

## 6. 業務規則（Business Rules）

| 規則編號 | 規則描述 |
|----------|----------|
| BR-01 | 密碼比對相符時方視為驗證通過。 |
| BR-02 | 密碼錯誤時不得進入服務選單。 |
| BR-03 | 提款金額不得為負數。 |
| BR-04 | 僅當帳戶餘額 ≥ 提款金額時，方可出鈔。 |
| BR-05 | 餘額不足時，帳戶餘額須維持原值且不出鈔。 |
| BR-06 | 提款成功後帳戶餘額 = 原餘額 − 提款金額。 |
| BR-07 | 未通過密碼驗證前不得執行提款。 |
| BR-08 | 結束服務後須退卡並清除當前交易狀態。 |

## 7. 使用案例規格（Use Case）

### UC-01 提款

| 項目 | 內容 |
|------|------|
| 使用案例名稱 | ATM 提款 |
| 主要參與者 | 持卡人（使用者 A） |
| 前置條件 | 持卡人持有效提款卡；ATM 正常運作 |
| 觸發條件 | 持卡人插入提款卡 |
| 後置條件（成功） | 現金出鈔完成、帳戶扣款完成、退卡 |
| 後置條件（失敗） | 帳戶餘額不變、不出鈔或未解鎖服務 |

**主要成功情境步驟**

1. 持卡人插入提款卡。
2. 系統要求輸入密碼。
3. 持卡人輸入正確密碼。
4. 系統解鎖服務選單。
5. 持卡人選擇提款 1000 元。
6. 系統檢查餘額並出鈔 1000 元。
7. 系統扣款 1000 元。
8. 持卡人結束服務，系統退卡。

**例外情境**

| 例外編號 | 情境 | 系統反應 |
|----------|------|----------|
| E-1 | 密碼錯誤 | 顯示「密碼錯誤」，不解鎖服務 |
| E-2 | 餘額不足 | 顯示「餘額不足」，不出鈔、餘額不變 |
| E-3 | 未驗證即提款 | 阻擋交易並回報密碼未通過驗證 |

## 8. 驗收標準（BDD 情境對應）

以下情境以繁體中文 Gherkin 撰寫，作為驗收依據。

### 8.1 正向情境 — 密碼正確且餘額充足時成功提款

| 步驟類型 | 內容 |
|----------|------|
| 假設 | 帳戶餘額為 5000 元，密碼為 "1234" |
| 當 | 使用者 A 輸入密碼 "1234" |
| 那麼 | 密碼驗證通過並可選擇服務 |
| 當 | 使用者 A 選擇提款 1000 元 |
| 那麼 | 提款機提供 1000 元現金 |
| 而且 | 帳戶餘額為 4000 元 |
| 當 | 使用者 A 選擇結束服務 |
| 那麼 | 提款機退出提款卡 |

### 8.2 反向情境 — 密碼錯誤

| 步驟類型 | 內容 |
|----------|------|
| 當 | 使用者 A 輸入密碼 "9999" |
| 那麼 | 密碼驗證失敗 |
| 而且 | 顯示錯誤訊息「密碼錯誤」 |
| 而且 | 無法選擇服務 |

### 8.3 反向情境 — 餘額不足

| 步驟類型 | 內容 |
|----------|------|
| 假設 | 帳戶餘額為 500 元 |
| 當 | 使用者 A 選擇提款 1000 元 |
| 那麼 | 顯示錯誤訊息「餘額不足」 |
| 而且 | 帳戶餘額維持 500 元 |
| 而且 | 提款機未提供現金 |

### 8.4 邊界情境（場景大綱）

| 餘額（元） | 提款金額（元） | 預期結果 | 剩餘餘額（元） |
|------------|----------------|----------|----------------|
| 5000 | 1000 | 成功 | 4000 |
| 1000 | 1000 | 成功 | 0 |
| 800 | 1000 | 餘額不足 | 800 |

## 9. 資料需求

| 資料項目 | 型別 | 說明 | 限制 |
|----------|------|------|------|
| 卡號（cardNumber） | 字串 | 提款卡識別碼 | 對應唯一帳戶 |
| 帳號（accountId） | 字串 | 帳戶識別碼 | 不可為空 |
| 密碼（pin） | 字串 | 帳戶密碼 | 4–6 位數字 |
| 餘額（balance） | 整數（元） | 帳戶可用餘額 | 不可為負 |
| 提款金額（amount） | 整數（元） | 單筆提款金額 | 不可為負、須 ≤ 餘額 |

## 10. 系統架構對應（DDD 分層）

| 分層 | 元件 | 職責 |
|------|------|------|
| Domain（領域層） | `Account`、`Money`、`Pin` | 封裝密碼驗證、提款與扣款規則 |
| Domain（介面） | `AccountRepository`、`CashDispenser` | 定義儲存與出鈔抽象（DIP） |
| Application（應用層） | `AtmService` | 協調插卡、驗證、提款、退卡用例 |
| Infrastructure（基礎設施層） | `InMemoryAccountRepository`、`SimpleCashDispenser` | 實作儲存與出鈔設備 |
