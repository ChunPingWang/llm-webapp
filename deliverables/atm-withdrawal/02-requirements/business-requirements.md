## 1. 文件目的

本文件旨在定義「富邦銀行 ATM 提款服務」之業務需求，明確描述使用者於自動櫃員機（ATM）進行插卡、密碼驗證、提款與退卡之完整流程，作為開發、測試與驗收之共同依據。

---

## 2. 專案背景與範圍

### 2.1 背景

為提供持卡人便捷、安全的現金提領服務，富邦銀行建置 ATM 提款系統。系統須確保交易安全（密碼驗證）與帳務正確（餘額扣款一致性）。

### 2.2 範圍（In Scope）

| 項目 | 說明 |
|------|------|
| 插卡與讀卡 | 使用者插入提款卡並辨識帳戶 |
| 密碼驗證 | 驗證使用者輸入之密碼是否正確 |
| 服務選單 | 密碼驗證通過後提供服務選項 |
| 提款交易 | 提領指定金額並扣除帳戶餘額 |
| 出鈔 | 提款機提供對應現金 |
| 結束服務與退卡 | 使用者結束服務並退出提款卡 |

### 2.3 範圍外（Out of Scope）

- 跨行轉帳、存款、餘額查詢、變更密碼等其他 ATM 功能。
- 實體鈔票庫存管理與補鈔作業。
- 網路銀行與行動銀行整合。

## 5. 業務流程說明

### 5.1 主要流程（Happy Path）

1. 使用者 A 於富邦銀行 ATM 插入提款卡。
2. 系統讀取卡片並要求輸入密碼。
3. 使用者輸入密碼，系統驗證正確。
4. 系統顯示服務選單，允許選擇服務。
5. 使用者選擇「提款 1,000 元」。
6. 系統確認帳戶餘額充足。
7. 提款機提供 1,000 元現金。
8. 系統於帳戶扣除 1,000 元。
9. 使用者選擇結束服務，系統退出提款卡。

### 5.2 流程圖（文字描述）

```
[插入提款卡] → [輸入密碼]
       │
       ├─ 密碼錯誤 → [顯示「密碼錯誤」] → [不進入服務選單]
       │
       └─ 密碼正確 → [顯示服務選單] → [選擇提款金額]
                              │
                              ├─ 餘額不足 → [顯示「餘額不足」] → [不出鈔]
                              │
                              └─ 餘額充足 → [出鈔] → [扣款] → [結束服務] → [退卡]
```

## 6. 功能性需求

| 需求編號 | 需求名稱 | 描述 | 優先級 |
|----------|----------|------|--------|
| FR-01 | 插卡辨識 | 系統應能讀取提款卡並辨識對應帳戶 | 高 |
| FR-02 | 密碼驗證 | 系統應驗證使用者輸入之 4 位數密碼 | 高 |
| FR-03 | 服務選單控制 | 僅在密碼驗證通過後開放服務選單 | 高 |
| FR-04 | 提款交易 | 系統應處理提款請求並判斷餘額 | 高 |
| FR-05 | 餘額扣款 | 提款成功後應正確扣除帳戶餘額 | 高 |
| FR-06 | 出鈔控制 | 提款成功後提款機應提供對應現金 | 高 |
| FR-07 | 結束與退卡 | 使用者結束服務後應退出提款卡 | 高 |
| FR-08 | 錯誤提示 | 密碼錯誤或餘額不足時顯示明確訊息 | 高 |

## 7. 非功能性需求

| 需求編號 | 類別 | 描述 |
|----------|------|------|
| NFR-01 | 安全性 | 密碼須以值物件封裝，不得明碼記錄於日誌 |
| NFR-02 | 一致性 | 出鈔與扣款須維持帳務一致，不得部分成功 |
| NFR-03 | 可維護性 | 系統採 DDD 分層與 SOLID 原則設計 |
| NFR-04 | 可測試性 | 交易邏輯須具備單元測試與 BDD 情境覆蓋 |
| NFR-05 | 品質門檻 | 測試涵蓋率須 ≥ 80% |
| NFR-06 | 語言 | 使用 Java 21 語法實作 |

## 8. 業務規則

| 規則編號 | 規則描述 |
|----------|----------|
| BR-01 | 密碼必須為 4 位數字，否則視為無效輸入 |
| BR-02 | 密碼錯誤時，不得進入服務選單，且不得異動餘額 |
| BR-03 | 提款金額不得超過帳戶餘額，否則交易失敗且不出鈔 |
| BR-04 | 提款成功時，扣款金額須等於出鈔金額 |
| BR-05 | 帳戶餘額不得為負數 |
| BR-06 | 金額不得為負數 |
| BR-07 | 結束服務後，卡片必須退出 |

## 10. 驗收情境（BDD）

驗收情境採用 Gherkin（繁體中文）撰寫，涵蓋正向與反向條件。

### 10.1 情境對照表

| 情境編號 | 類型 | 密碼 | 初始餘額 | 提款金額 | 預期結果 | 最終餘額 |
|----------|------|------|----------|----------|----------|----------|
| SC-01 | 正向 | 正確 | 5,000 | 1,000 | 成功、出鈔 | 4,000 |
| SC-02 | 反向 | 錯誤 | 5,000 | — | 密碼錯誤 | 5,000 |
| SC-03 | 反向 | 正確 | 500 | 1,000 | 餘額不足 | 500 |
| SC-04 | 邊界 | 正確 | 1,000 | 1,000 | 成功、出鈔 | 0 |

### 10.2 Gherkin 驗收腳本

```gherkin
# language: zh-TW
# 檔名: atm_withdrawal.feature
功能: 富邦銀行 ATM 提款服務

  背景:
    假設 使用者 "A" 持有富邦銀行提款卡
    而且 該卡片綁定的帳戶密碼為 "1234"

  場景: 正向 - 密碼正確且餘額充足時成功提款
    假設 使用者 "A" 的帳戶餘額為 5000 元
    當 使用者 "A" 插入提款卡
    而且 使用者 "A" 輸入密碼 "1234"
    那麼 系統應允許使用者選擇服務
    當 使用者 "A" 選擇提款 1000 元
    那麼 提款機應提供 1000 元現金
    而且 使用者 "A" 的帳戶餘額應為 4000 元
    當 使用者 "A" 選擇結束服務
    那麼 系統應退出提款卡

  場景: 反向 - 密碼錯誤時無法進入服務選單
    假設 使用者 "A" 的帳戶餘額為 5000 元
    當 使用者 "A" 插入提款卡
    而且 使用者 "A" 輸入密碼 "9999"
    那麼 系統應顯示 "密碼錯誤" 訊息
    而且 系統不應允許使用者選擇服務

  場景: 反向 - 餘額不足時提款失敗
    假設 使用者 "A" 的帳戶餘額為 500 元
    當 使用者 "A" 插入提款卡
    而且 使用者 "A" 輸入密碼 "1234"
    那麼 系統應允許使用者選擇服務
    當 使用者 "A" 選擇提款 1000 元
    那麼 系統應顯示 "餘額不足" 訊息
    而且 提款機不應提供現金
    而且 使用者 "A" 的帳戶餘額應為 500 元
```

## 11. 領域模型設計

### 11.1 DDD 分層

| 分層 | 元件 | 職責 |
|------|------|------|
| Domain | `Account`（聚合根） | 密碼驗證、扣款業務規則 |
| Domain | `Money`、`Pin`（值物件） | 封裝金額與密碼，保證不變性 |
| Domain | `AccountRepository`、`CashDispenser`（介面） | 定義抽象契約（DIP） |
| Application | `AtmService` | 協調提款用例 |
| Application | `WithdrawalResult`（列舉） | 表達交易結果狀態 |
| Infrastructure | `InMemoryAccountRepository`、`SimpleCashDispenser` | 具體技術實作 |

### 11.2 SOLID 對應

| 原則 | 落實方式 |
|------|----------|
| SRP | 各類別職責單一（帳戶、金額、服務分離） |
| OCP | 透過介面擴充儲存與出鈔實作 |
| LSP | 實作類別可替換介面而不破壞行為 |
| ISP | `AccountRepository` 與 `CashDispenser` 介面精簡 |
| DIP | 應用層依賴抽象介面，非具體實作 |

### 11.3 領域物件關聯（文字描述）

```
AtmService ──依賴──> AccountRepository（介面）
AtmService ──依賴──> CashDispenser（介面）
AccountRepository ──管理──> Account（聚合根）
Account ──持有──> Pin（值物件）、Money（值物件）
```

## 12. 例外與錯誤處理

| 錯誤情境 | 系統回應 | 帳務影響 |
|----------|----------|----------|
| 密碼格式錯誤（非 4 位數字） | 拋出輸入例外 | 無 |
| 密碼驗證失敗 | 顯示「密碼錯誤」 | 無 |
| 餘額不足 | 顯示「餘額不足」，不出鈔 | 無 |
| 卡片無法辨識 | 回傳「卡片無效」 | 無 |
| 金額為負數 | 拋出參數例外 | 無 |

## 14. 假設與限制

### 14.1 假設

- 每張提款卡對應單一帳戶。
- 提款機現金充足，不考慮鈔票庫存不足情境（本版本範圍外）。
- 密碼固定為 4 位數字。

### 14.2 限制

- 本版本僅實作提款用例，其他 ATM 功能不含。
- 儲存層採記憶體實作，未串接實體核心銀行系統。
