## 1. 文件目的

本文件旨在定義「富邦銀行 ATM 提款服務」之業務需求，作為需求方、開發團隊與測試團隊之共同依據。內容涵蓋業務目標、範圍、角色、業務流程、功能與非功能需求、業務規則、驗收準則，並對應行為驅動開發（BDD）之 Gherkin 情境與領域驅動設計（DDD）之分層架構。

---

## 2. 業務背景與目標

### 2.1 背景

富邦銀行持卡人需透過 ATM 進行自助提款。系統須在確保安全（密碼驗證）與一致性（帳戶扣款與現金吐鈔同步）的前提下，提供快速、正確的提款服務。

### 2.2 業務目標

| 編號 | 目標 | 衡量指標 |
|------|------|----------|
| G-01 | 提供安全的身分驗證 | 密碼錯誤者 100% 無法進入服務選單 |
| G-02 | 確保提款交易一致性 | 扣款與吐鈔不一致率為 0% |
| G-03 | 提升交易成功率 | 條件符合之提款成功率 ≥ 99.9% |
| G-04 | 明確的錯誤提示 | 所有失敗情境皆顯示對應訊息 |

## 3. 範圍

### 3.1 範圍內（In Scope）

- 插卡與退卡流程
- 密碼驗證

## 6. 業務流程

### 6.1 主要流程（Happy Path）

1. 使用者 A 插入提款卡。
2. 系統要求輸入密碼。
3. 使用者 A 輸入密碼，密碼正確。
4. 系統顯示服務選單。
5. 使用者 A 選擇提款並輸入金額 1,000 元。
6. 系統檢查現金庫存充足且帳戶餘額充足。
7. 系統於帳戶扣款 1,000 元。
8. ATM 吐出 1,000 元現金。
9. 使用者 A 選擇結束服務。
10. ATM 退出提款卡，回到待機畫面。

### 6.2 流程圖（文字流程）

```
[插卡] → [輸入密碼] → 密碼正確? 
   ├─ 否 → [顯示「密碼錯誤」] → [結束]
   └─ 是 → [顯示服務選單] → [選擇提款/輸入金額]
              → 現金充足?
                 ├─ 否 → [顯示「現金庫存不足」] → [結束]
                 └─ 是 → 餘額充足?
                            ├─ 否 → [顯示「餘額不足」] → [結束]
                            └─ 是 → [扣款] → [吐鈔] → [結束服務] → [退卡]
```

| 編號 | 功能名稱 | 說明 | 優先級 |
|------|----------|------|--------|

| FR-02 | 密碼驗證 | 驗證輸入密碼是否與綁定密碼相符 | 高 |

| FR-04 | 提款交易 | 輸入金額並執行提款 | 高 |
| FR-05 | 帳戶扣款 | 交易成功後於帳戶扣除對應金額 | 高 |
| FR-06 | 現金吐鈔 | ATM 吐出對應金額現金 | 高 |
| FR-07 | 結束服務並退卡 | 交易完成後退出提款卡並回待機 | 高 |
| FR-08 | 錯誤訊息提示 | 顯示密碼錯誤／餘額不足／現金不足訊息 | 高 |

## 8. 業務規則

| 編號 | 規則描述 |
|------|----------|
| BR-01 | 密碼須為 4 位數字，且須與卡片綁定密碼完全相符方可進入服務選單。 |
| BR-02 | 密碼錯誤時，不得顯示服務選單，且帳戶餘額不得變動。 |
| BR-03 | 提款前須先檢查 ATM 現金庫存是否充足。 |
| BR-04 | 提款金額不得超過帳戶餘額，否則交易失敗且不扣款。 |
| BR-05 | 交易須確保「先扣款成功後才吐鈔」，任一環節失敗不得造成帳戶與現金不一致。 |
| BR-06 | 金額不可為負數。 |
| BR-07 | 交易完成後須退出提款卡並回到待機畫面。 |

## 10. 使用案例（Use Cases）

### UC-01：成功提款

| 項目 | 內容 |
|------|------|
| 主要角色 | 持卡人（使用者 A） |
| 前置條件 | 已插卡、帳戶餘額 ≥ 提款金額、現金庫存充足 |
| 主要流程 | 輸入正確密碼 → 選擇提款 → 扣款 → 吐鈔 → 退卡 |
| 後置條件 | 帳戶餘額減少、現金庫存減少、卡片退出 |

| 項目 | 內容 |
|------|------|

| 後置條件 | 不顯示選單、餘額不變 |

### UC-03：餘額不足

| 項目 | 內容 |
|------|------|
| 前置條件 | 已插卡、密碼正確、餘額 < 提款金額 |
| 主要流程 | 選擇提款 → 顯示「餘額不足」 |
| 後置條件 | 不吐鈔、餘額不變 |

### UC-04：現金庫存不足

| 項目 | 內容 |
|------|------|
| 前置條件 | 已插卡、密碼正確、庫存 < 提款金額 |
| 主要流程 | 選擇提款 → 顯示「現金庫存不足」 |
| 後置條件 | 不吐鈔、餘額不變 |

## 11. 驗收準則（對應 BDD 情境）

以下 Gherkin 情境作為正式驗收依據，需求方與測試團隊以此為準。

```gherkin
# language: zh-TW
# 檔名: atm_withdraw.feature
功能: 富邦銀行 ATM 提款服務

  背景:
    假設 使用者 "A" 持有富邦銀行提款卡 "1234-5678"
    而且 該卡片綁定的密碼為 "4321"
    而且 使用者 "A" 的帳戶餘額為 5000 元
    而且 ATM 現金庫存為 10000 元

  場景: 密碼正確並成功提款 1000 元
    假設 使用者 "A" 已插入提款卡
    當 使用者 "A" 輸入密碼 "4321"
    那麼 系統應顯示服務選單
    當 使用者 "A" 選擇提款服務並輸入金額 1000 元
    那麼 ATM 應吐出 1000 元現金
    而且 使用者 "A" 的帳戶餘額應為 4000 元
    當 使用者 "A" 選擇結束服務
    那麼 ATM 應退出提款卡

  場景: 密碼錯誤無法進入服務選單
    假設 使用者 "A" 已插入提款卡
    當 使用者 "A" 輸入密碼 "0000"
    那麼 系統應顯示 "密碼錯誤" 訊息
    而且 系統不應顯示服務選單

  場景: 帳戶餘額不足無法提款
    當 使用者 "A" 選擇提款服務並輸入金額 1000 元（餘額 500）
    那麼 系統應顯示 "餘額不足" 訊息
    而且 ATM 不應吐出現金

  場景: ATM 現金不足無法提款
    當 使用者 "A" 選擇提款服務並輸入金額 1000 元（庫存 500）
    那麼 系統應顯示 "現金庫存不足" 訊息
    而且 ATM 不應吐出現金
```

### 驗收條件對照表

| 情境 | 對應使用案例 | 業務規則 | 預期結果 |
|------|--------------|----------|----------|
| 成功提款 1000 元 | UC-01 | BR-01, BR-03, BR-04, BR-05 | 吐鈔、餘額 4000、退卡 |
| 密碼錯誤 | UC-02 | BR-01, BR-02 | 顯示「密碼錯誤」、不顯示選單 |
| 餘額不足 | UC-03 | BR-04 | 顯示「餘額不足」、不吐鈔 |
| 現金不足 | UC-04 | BR-03 | 顯示「現金庫存不足」、不吐鈔 |

## 14. 風險與對策

| 風險 | 影響 | 對策 |
|------|------|------|
| 扣款成功但吐鈔失敗 | 帳戶不一致 | 先扣款後吐鈔並確保原子性（BR-05） |
| 密碼暴力破解 | 安全風險 | 後續版本加入重試上限 |
| 現金庫存耗盡 | 交易中斷 | 維運監控與即時補鈔 |
